# FDE Bridge

Headless API bridge that lets Claude Code programmatically drive Palantir AI FDE sessions running in Chrome.

## System Architecture

```
Claude Code
  └─ fde_run() MCP tool (mcp-tool/dist/index.js, StdioServerTransport)
       └─ HTTP POST 127.0.0.1:27182  [X-FDE-Token header]
            └─ native-host/server.js  (Node.js, http.createServer on 127.0.0.1)
                 └─ native messaging  (stdin/stdout, 4-byte LE length + JSON)
                      └─ extension/background.js  (Chrome MV3 service worker)
                           └─ chrome.tabs.sendMessage
                                └─ extension/content-script.js  (injected into Foundry tab)
                                     └─ DOM: inject prompt → wait for response → return text
```

## Component Responsibilities

| File | Runtime | Role |
|---|---|---|
| `mcp-tool/index.ts` | Claude Code process | MCP server exposing `fde_run` tool; HTTP client |
| `native-host/server.js` | Persistent Node.js process | Owns TCP socket; bridges HTTP ↔ native messaging |
| `extension/background.js` | Chrome SW (ephemeral MV3) | Routes native messages → content script; keepalive |
| `extension/content-script.js` | Foundry tab (always-on) | DOM interaction, config apply, response detection |
| `extension/popup.html/.js` | Chrome popup | Status display |

## Native Messaging Protocol

Framing: `[4-byte LE uint32 length][UTF-8 JSON payload]`. Max 1 MB per message.

**Host → Extension:**
- `{ type: "handshake", token: string }` — on server startup, syncs auth token
- `{ type: "request", requestId: string, payload: { prompt, new_session, config_override, timeout_ms } }` — FDE execution request
- `{ type: "config_sync", config: object }` — push config to extension storage
- `{ type: "pong" }` — keepalive response

**Extension → Host:**
- `{ type: "ping" }` — keepalive from 24s alarm
- `{ type: "tab_status", tabId: number, status: "ready" }` — Foundry tab registered
- `{ type: "response", requestId: string, result?: { response: string }, error?: { code, message } }` — FDE result

## Auth

- 32-byte hex token generated by `install.sh`, stored at `~/.config/fde-bridge/token` (mode 600)
- Required as `X-FDE-Token` header on all HTTP requests
- HTTP server binds to `127.0.0.1` only (not `0.0.0.0`)
- Never commit the token; `.mcp.json` leaves `FDE_BRIDGE_TOKEN` empty — the MCP tool reads from file

## Service Worker Keepalive (MV3)

Chrome MV3 SWs terminate after ~30s of inactivity. `chrome.alarms` with `periodInMinutes: 0.4` (every 24s) keeps the SW event loop alive and re-establishes the native port if it dropped. The port is also re-checked defensively at the start of every message handler.

## React Input Injection (content-script.js)

Three-layer approach for React 18 contenteditable inputs:
1. `document.execCommand('insertText', false, text)` — preferred; sets `nativeEvent.data` for React's synthetic event system
2. `new InputEvent('input', { inputType: 'insertText', data: text, bubbles: true })` — fallback
3. React fiber walk (`__reactFiber*` key) to call `onChange` directly — nuclear fallback

For `<textarea>`: use `HTMLTextAreaElement.prototype` native value setter, not `element.value = ...`.

## Response Completion Detection (content-script.js)

- `MutationObserver` on `document.body` watching attribute changes (`disabled`, `aria-disabled`, `class`)
- **Primary signal:** send button transitions disabled → enabled
- **Secondary:** DOM stability — no mutations for 2000ms
- **Hard timeout:** `timeout_ms` (default 5 min); rejects with `{ code: 408, partial: string }`

## DOM Selectors — FIELD VERIFY

These selectors in `content-script.js` are best-guess patterns. Open DevTools on your Foundry FDE page and confirm before relying on automation:

```
$('[data-testid]')            — enumerate all testid elements
$('[contenteditable]')        — find the chat input
$('button[disabled]')         — find the send button when it's disabled
```

Key things to verify:
- Is the chat input a `contenteditable` div or a `<textarea>`?
- What `data-testid` or `aria-label` values are on the send button, tool toggles, and new-session control?
- Does the send button use the HTML `disabled` attribute or `aria-disabled="true"`?
- What is the exact URL path for the FDE page? (used for `new_session` navigation fallback)

## Stable Extension ID

The RSA public key in `extension/manifest.json` gives a fixed extension ID: `mmoolbcfmhiijogpplbcepihjpacmfaj`. This is baked into `native-host/install.sh` and `native-host/com.palantir.fde.bridge.json`. Do not regenerate the key.

## Config File

Optional at `~/.config/fde-bridge/fde-config.json`. Schema: `config/fde-config.schema.json`. Applied by the content script at session start to set tool toggles and approval mode. Can also be passed per-call via `config_override` in the HTTP body.

## Common Issues

| Symptom | Cause | Fix |
|---|---|---|
| HTTP 503 | Extension not connected | Open a Foundry tab; wait ~5s |
| HTTP 404 | No Foundry tab found | Navigate to AI FDE page in Chrome |
| HTTP 401 | Wrong token | Check `FDE_BRIDGE_TOKEN` env or `~/.config/fde-bridge/token` |
| Native host not found | Manifest not installed | Re-run `install.sh`; confirm extension ID |
| Prompt not injected | Wrong selector | Verify `SELECTORS.chatInput` in DevTools |
| Response never completes | Wrong send button selector | Verify `SELECTORS.sendButton` disabled state |

Logs: `tail -f ~/.config/fde-bridge/server.log`
